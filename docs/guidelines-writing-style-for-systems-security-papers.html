4 Guidelines: Writing Style for Systems Security Papers

Through my various research collaborations, my writing style has been influenced by different mentors. Some of these styles were conflicting, requiring me to adapt. But my current the writing style that I use has been heavily inspired by my advisor Prof.¬†N Asokan, who was extremely patient while I made terrible writing mistakes. While there is still a long way to improve my writing, over time, I got a sense of how to write papers as expected by my advisor. The reason I rely and advocate this heavily is that (a) this is well-structure, systematic and easy to follow, and (b) responses from reviewers about writing have been mostly positive (including in some cases praises for combining a lot of context well as in our SoK paper which won the distinguished paper award at Oakland). In this blog post, I want to document the writing style in our group which I hope will be useful for others. Also, I will give concrete examples from existing papers to illustrate some aspects. For all the papers I have written after joining SSG, we have followed a more or less fixed template. You can see the pattern across the different papers clearly: paper1, paper2, ‚Ä¶ I found this standard template to be very systematic and easy to following. And generally, any new project which fits most of the themes (designing a new ML system (e.g., Laminator) or designing a new defense (e.g., grove, espresso), I can blindly follow the same template and get some decent systematic structure for the paper which is easy to follow. I want to discuss these below. Note that there are definetely variants of the template or cases where the template did not fit well (e.g., SoK does not have a problem statement) but they are usually within some reasonably minimal deviations from template. I also include a lot of my personaly style preferences for paper writing and this blog leans on that at several points. Structure We use overleaf and for all any new project, I include the following files as part of the basic template (naming is numbered so that it is easy to see since it is ordered according to the sections): 0macros.tex: contains all the macros 0abstract.tex: Indicates the text for abstract 1introduction.tex 2background.tex 3problem.tex 4approach.tex 5setup.tex: experimental setup: datasets, models, metrics, baselines (which could be described in background section but here we only highlight what is it we compare with) 6evaluation.tex: results and showing how your design meets all the described requirements laid out in 3problem.tex 7discussion.tex Writing: Start with the structure of the paper Then include the possible sections to include and add topic sentences for each Create a macro within which we indicate a high level description of what should be included. Then start writing the text and later switch off the macro. Note: In case of limited space, you can consider using the symbol ¬ß to replace ‚Äúsection‚Äù. 0macros.tex: Macros I mention some standard libraries and function calls I used for most papers which is useful and can be copied for all subsequent projects. gratutous use of synonyms: when using a word, always think if this will confuse the reader; technical words should be fixed; no synonyms; general english terms are ok 0abstract.tex: Abstract One line each: a general statement of the issue to be solved. why is it important and challenging a sentence motivating your goal of this work: ‚ÄúHence, we need a defense which protects against‚Ä¶‚Äù Indicate the desiderate or requirements for an ideal system: Ideally, a defense should be‚Ä¶ Limitations of prior work: Could be a simple statement like ‚ÄúNone of the prior works meet all requirements‚Äù [No need to go into details in teh abstract; we will add more context in intro and further more in problem statement] All the above can be in one paragraph. Now we switch contexts, and indicate our contributions. Present our framework, and text to indicate what is unique with it (the first framework which is robust‚Äù the general ideal on why it works and addressing the challenges faces by prior works results (including some numbers XX% aaccuracy on YY)‚Ä¶and indicating that it satisfiess all requirements. One paragraph or two paragraphs: Ideally, I try to keep it one paragraph. But sometimes the problem description adn limitations of prior work take up space. In which case, I spearate our contributions and its result summary in a second paragrpah. Both are fine. If it is getting too long, split into two paragraphs. 1introduction.tex: Introduction one paragraph for each: general introduction to the area, current problem, challenges to solve the problem, ideal way to solve the problem (specify the requirements/desiderata), limitations of existing work, and our idea/intuition and brief approach along with the contributions (in bullet points) Contributions (No need for a specific subheading called contributions.): In some papers, you will find very long contributions which span paragraphs explaining how they solve the problem. This makes it hard for people (like me) who sometimes want o skim the paper and get a sense of the approach and their results in a quick glance. This requires me to read more than I want to‚Ä¶hence, going more towards conciseness, the constributions should be always be specific with forward pointers to the sections (do not waste space with an extra paragrpah saying the outline of the paper‚Ä¶.e.g., We will discuss blabla in Section A, blublu in section B, cluclu in Section C, ‚Ä¶). Also, not many people tend to read them since by looking at the headings in the papers, you can get a sense of what it will include. Style Statement: Generally, problem statement can be small and may not require subsection. See my note on subsection in Misc towards the end. If the subsections are small, consider using to highlight the subheading. You might have the following subheading under problem statement: Threat Model/Adversary Description/Adversary Model System Model Requirements/Desiderata Limitations of Prior works (in light of the requirements) Challenges (set of challenges for building a system that you want to address in order to satisfy a requirement) Threat Model/Adversary Description/Adversary Model This should include assuptions about the adversary. In some cases, you can have a defender, or model builder, or prover, with their own set of assumptions. We can include their, goals, assumptions, access, knowledge (similar to adversary description) for describing the defender. Requirements/Desiderata Requirements are more strict and should be satified ideally; Desiderate is more like a general guidelines that should be followed.. This can be done to systematically indicate bullet points for challenges or limitations as well. For a lot of these, I generally use some latex enumerate type libraries to identify multiple requirements, challenges, which helps me refer to them later in the paper. Here are some example requirements I have used in prior work: effective (to reduce the attack success rate) efficient (incurs a low overhead) applicable/general (can be used for differrent cases without modification) Non-invasive (does not require making any changes) extensible (allows including more risks, parameters, etc.) robust (resist evasion by adversaries) utility-preserving (maintains acceptable utility on non-adversarial data) Scalable (supports large numbers of provers/verifiers). Versatile (supports a wide variety of properties; requires minimal effort to add support for new properties). Rest of the paper, should following the requirements. In toerhh words, the approach and evaluation section, shoudl constatly keep referring to how your design choices help satisfy some requirement or explicitly have subsections to either indicate the design choice to satisfy the requirement or to evaluate the system for a particular requirement. Note that some of the requirements may be a discussion rather than empirical evaluation (see non-invasive and general requirements for combining exisiting defenses; or scalable or versatile properties in laminator. Limitation of Prior work Firstly, a thourough coverage of related work is important. Further subscribe to arxiv daily digest and go voer them whenever the mail comes religiously. This helps to identify related works and even interesting unrelated works which could be future projects. I have had many such cases where I remember seeing a paper from the igest which turned out to be a related work for a new project. Systematically cover all the different papers and describe them; Since you have identified a set of requirements for an ideal system. (e.g., defense, framework), this helps to evaluate others with respect to those requirements. Have a table sumarizing how different papers satisfy different works. You can use checkmark or full circle for requirements being satisfied, or cross or empty circle for not satisfies (see wasysm or pifont libraries). Alternatively, partially sastified requirement can also be used but it has to be justified well‚Ä¶.what counts as being partially satified? To decide whether or not empirically evaluate them, consider the following: if it is a SOTA for a given requirement, definitely use it for comparison any other non-SOTA can be ignore by citing the SOTA only if the SOTA has indicated an explicit comparison with other snad shown that they outperform them If there are papers which are not evaluated by SOTA, you might have to add that benchmark. If for most requirement, a defense does not satify, you can discount them saying they do not meet the requirements. But check if they are SOTA in any specific requiremtn and may need a comparison as indicated in the first point. Using LLMs: My writing has improved in terms of less errors (grammar, spellings etc) after using LLMs. I have found LLMs can be useful when I have a garbled sentence but need to make ti more clear and precisie. Hence, asking it to improve text and making it concise generally does a decent job. However, using LLMs for entire paragraphs and sections is too much specially that I have to go voer the output to make sure the model did not generate some incorrect stuff. This is time consuming. So i tend to use it for limited cases for some specific sentences. Check for Logical Consistency among Statements For every subsequent statement, make it it flows well wrt to the previous one. If there is any abrupt changes or awkward reading, that is a sign to consider removing, rephrasing or adding more context to make the flow better. This is a constant check that has to be done for every statement you write. Backup Every Statement This is very important. Never make statements that you cannot verify or backup. It is empty statements and mean nothing. There are three ways to do it (should be considered in order when you are trying to back up your statements): refer to prior works (citation to say that you statement was verifier by some other works empirically) logically argue (there could be cases where some deductions can be straightforward and do not need an evlauation; for instance, their appropach uses fine-tning over ABC and DEF requiring XYZ GPUs and hence, inefficient for practical use). Empirically backing up: If none of the above work, you have to resort to setting up an experiment and evaluating it. For instance, in our SoK, we made a statement that ‚Äúoverfitting and memorization can occur together or indepdently‚Äù While there was some citations previously, there was nothing concrete to verify the different cases. So, we had a small experiment (separate from that in the main paper) to validate this. Note that this had some connection to the discussion at hand since overfitting and memorization were critical to the paper, and we had space. If your statement is not critically important and /or limited space, you can resort to the results in teh appendix and state in the main body ‚Äúwe conduct a small experiments to verufy this which is discussed in Appendix A‚Äù. Consistency Never use synonyms for any technical terms. If you have defined a term in the beginning, make sure the rest of the paper also uses it. Synonyms while improves general english writing is likely to confuse readers in technical writing. So ensure the same terms are consistently used everywhere. 4.1 Post-publication Prcoess

Work on the code and make it publicly available on the group‚Äôs github page. Make one single clean push with a new repo so no commit history is visible. See XX on guidelines to write code.

Push the camera ready version on arxiv with an indication of where it is published in comments.

Work on a short blog (if required) highlighting the problem, key results, and main takeaway.

Update the website with the new publication in the correct format and all the appropriate links (including to the blog post). Make a post for X and LinkedIN using the official group handles (and not personal handles) sharing the work. You can use the following template:

Our paper titled "XX" has been accepted to "YY"

**Authors**: X, Y, Z

**Problem**: <one line description>

Approach: <one line description>

Results + Takeaways: <one line description: We show that...This highliths the importance of...>

Start working on on the presentation (see XXX) and get feedback and finalize it. We make a master‚Äôs slide deck which is a long form presentation covering all parts of the paper. We then create a copy of the master deck to specific cases depending on the time limit as required.

4.1.1 Process

Once you give me a draft of some sections or the paper‚Ä¶I will go over and leave comments and make changes‚Ä¶Ideally, I will do this online with you so you can see how to make the changes and write the papers in the hope that you will pick it up and can do it independently in the future with minimal supervision. These would be long meetings to go over the paper together but use these initial meetings as a teaching session to write papers well.

Ideally, start by giving me the abstract and introduction. While many people prefer to write it at the end, writing this at the beginning gives an idea about how to position rest of the paper. Once we have finalized the storyline of the paper and agreed on common termiinplogy (do not use synonyms for technical words) you can update the rest of the text accordingly to rflect the introduction.

Generally, once I make edits and sign off on the changes, do not overwrite them unless you run the changes by me and I can approve. otherwise, there might be additional inconsistencies or clarifications which are missed.
